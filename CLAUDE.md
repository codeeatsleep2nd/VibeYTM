# VibeYTM

Apple Music-style YouTube Music desktop app.

## Tech Stack
- Tauri 2.x (Rust backend, WKWebView frontend)
- React 19 + TypeScript (custom UI)
- Vite (build tool)

## Commands
- `pnpm install` — install dependencies
- `pnpm tauri dev` — run in development mode
- `pnpm tauri build` — production build
- `pnpm typecheck` — TypeScript type check
- `cargo check` — Rust type check (run from src-tauri/)
- `cargo test` — Rust tests (run from src-tauri/)

## Dev Workflow
- After every code change that requires a restart to take effect (any Rust source under `src-tauri/`, anything under `scripts/inject/` injected into the YTM webview, `tauri.conf.json`, or `Cargo.toml`), automatically restart `pnpm tauri dev` without waiting for the user to ask. Standard pattern: `pkill -f "tauri dev" 2>/dev/null; pkill -f "VibeYTM" 2>/dev/null; sleep 1; pnpm tauri dev` (run in background).
- Frontend-only changes (`src/**/*.{ts,tsx,css}`) are picked up by Vite HMR — do NOT restart for those unless module-level state needs to be reset (e.g. changes to a module exporting mutable singletons).
- After restarting, verify the build came up cleanly via the task output before reporting work as done.

## Verification Discipline
- NEVER ask the user to debug, test, or verify a fix unless the action genuinely cannot be performed without human input (e.g. interactive system dialogs, listening to audio, judging visual aesthetics).
- For everything else, verify the fix yourself before declaring it done:
  - Read the dev-server task output at `/private/tmp/claude-501/.../tasks/<task-id>.output` for runtime logs (track changes, /next calls, queue updates, errors).
  - Inspect dumped YTM API responses at `/tmp/vibeytm-resp-*.json` to confirm what the backend actually received and parsed.
  - Run the full validation suite: `pnpm typecheck`, `pnpm test` (vitest), `cd src-tauri && cargo check`, `cargo test --lib`. Test counts are intentionally not pinned here — they grow with every PR; trust the runner output, not the docs.
  - Trace the previously-fixed code paths against the latest diff to confirm no regression of earlier bugs.
- When extra runtime visibility is needed, add a debug line via the bridge's `log()` ring (writes to `window.__VIBEYTM_DEBUG__`, surfaced by the Rust poller in dev-server output) — do NOT instruct the user to open WebView devtools and paste output.
- If something truly cannot be verified without the user (e.g. "does the lyric sync feel right by ear?"), say so explicitly: "I cannot verify X — please report what you observe" — instead of generically asking them to test.
- Before touching any file that has been the subject of repeated bug reports (especially `QueuePanel.tsx`, the bridge, and `playerApi`), re-read the full file and write down the invariants it depends on. Don't edit only the diff target.
- After each round of fixes, walk every previously-reported bug related to that area against the current code and confirm in the response that each prior fix is still in place — line numbers cited.

## Conflict Detection — ASK BEFORE DECIDING
Before implementing ANY new feature ask or bug-fix ask, scan whether it conflicts with an existing feature or established invariant. Examples that count as conflicts:
- The new ask reverses or weakens a rule documented in CLAUDE.md (e.g. "never show video thumbnails", "use real `<button>`, not `<div role=\"button\">`", "no `transform` on `ReloadOverlay` children").
- The new ask changes the cache lifetime of a value another feature depends on for correctness (e.g. "load lyrics from cache only" while another feature relies on remote refresh).
- The new ask changes the source of truth for state another component reads (e.g. switching artwork from the registry to a fresh IPC when the registry was specifically introduced to avoid that IPC).
- The new ask is the inverse of a fix shipped earlier in the same session (e.g. re-introducing always-on probes that were dropped to fix bridge saturation).

When a conflict is detected, **STOP and ask the user** — phrase the conflict explicitly ("X would undo Y from commit Z; want to proceed, refine the new ask, or revert Y?") and wait for their decision. Do NOT silently pick one side. Conflicts that look minor on the diff often reintroduce the very bug the prior fix closed; the user owns that trade-off.

## Visual fidelity to external products — VERIFY before designing
When the ask is "make UI match X" (Apple Music, Spotify, YouTube Music, iOS Now Playing, etc.), do NOT design from memory. Memory of a third-party UI is unreliable — button shapes drift, layouts get wrong from old screenshots in training data, and "I think it looks like…" produces ugly approximations that the user has to send you back to redo. Before writing any plan or component:

1. **Pull a real reference.** Use Chrome DevTools MCP (`new_page` → `take_screenshot` of the real product, e.g. music.apple.com), open the actual app side-by-side, or fetch the vendor's design-system page. Save the screenshot to `/tmp/` so it can be re-read this turn.
2. **Inventory what you see** — name every visible element, its position, shape, color, and approximate size. Write the inventory into the plan or a scratch doc; don't carry it only in your head.
3. **Don't assume position.** "Apple Music chrome" doesn't mean "moved to the top" unless the user says so. Ask which dimensions of the reference to mirror (visual treatment vs. layout vs. position) before committing to architecture-changing moves.
4. **Don't invent SF-Symbol glyphs from Unicode.** SF Symbols ≠ Unicode codepoints — `↻` is `arrow.clockwise` (reload), not `repeat`. Either ship inline SVG that matches the actual SF Symbol shape (verified against a reference), or use a vendored icon set whose mapping you've confirmed visually.
5. **Show the user a mockup before writing code.** When the redesign is non-trivial, paste an ASCII mockup or a labeled screenshot annotation and let them confirm direction. A wrong mockup is a 30-second correction; a wrong implementation is a multi-file revert.

If you skipped any of the above and the user pushes back with "that's not what X looks like," stop iterating on memory — go fetch a real screenshot before the next attempt.

## Search GitHub before implementing — REUSE BEFORE INVENT
Before designing or coding ANY non-trivial feature, search GitHub (and other registries — npm/crates.io/etc.) for existing implementations or open-source clones that already solve the same problem. Apple Music desktop wrappers, lyrics scrobblers, queue scrapers, IPC bridges — almost everything VibeYTM does has been done before, often by people who actually use the upstream service every day and have already worked around its quirks. Use these as references for:

1. **Visual fidelity** — desktop wrappers (e.g. `Alex313031/apple-music-desktop`, `wimpysworld/sidra`, `revblaze/AppleMusicUltra`) embed the real product UI, so their screenshots are ground truth and their style files often capture the design tokens.
2. **API behavior & quirks** — YTM bridge / Innertube clients (e.g. ytmusicapi, sigma67, th-ch/youtube-music) have already documented the response shapes and the edge cases.
3. **Implementation patterns** — when you'd otherwise hand-roll something (e.g. a queue dedupe, a media-key handler, a notification provider), check whether a small library or a referenced source-snippet already does it correctly.

Run `gh search repos`, `gh search code`, or a targeted web search BEFORE writing the plan — not after the user pushes back. Cite the references you used in the plan so future-you can re-verify. If no existing implementation matches, document what you searched for so the absence is recorded, not assumed.

## Architecture
- Two WebView model: visible React UI + hidden YouTube Music audio engine
- Event-driven: tokio broadcast bus connects all components
- Plugin-based integrations: each implements `Integration` trait
- PlayerState (Rust) is the single source of truth

## Versioning
- Bump the patch version (last number) in both `package.json` and `src-tauri/tauri.conf.json` with every commit that includes code changes
- Do NOT bump for docs-only commits (e.g. changes to .md files only)
- Example: 0.5.0 → 0.5.1 → 0.5.2 → ... → 0.5.10 → 0.5.11

## Release Process
- When creating a new GitHub release, always build the DMG first with `pnpm tauri build`
- Attach the DMG (`src-tauri/target/release/bundle/dmg/VibeYTM_*.dmg`) to the release
- Always include the macOS Gatekeeper notice at the top of release notes (the app is unsigned, so users must run `xattr -cr /Applications/VibeYTM.app` after installing)
- **Smoke-test the release DMG locally before publishing** for any change touching the poller, bridge JS, IPC handlers, or anything timing-sensitive. `pnpm tauri dev` is not sufficient — the dev path uses `debug_assertions`-gated code, different React (PROD vs DEV), and a runtime-injected Vite bundle, so behavior can diverge from a release DMG. Build the DMG, install to `/Applications`, and play through one or two tracks for ~30 s before cutting the GH release.
- For test-only DMGs (verifying a fix before publishing), append a `-dev` semver pre-release marker to the version (`<next-patch>-dev`, e.g. `1.6.1-dev`). Don't commit the `-dev` version. When the test is approved, revert to a clean `<next-patch>`, commit the actual fix with the version bump, then build + publish.

## Dev/release parity invariants (issue #flicker-may-2026)
- **`#[cfg(debug_assertions)]` / `#[cfg(not(debug_assertions))]` may only gate *diagnostic side effects* (logging, asserts, panics), never alternative implementations that contribute to state.** This codebase shipped a release-only branch in `webview_bridge/poller.rs` that scanned the whole bridge-debug ring every cycle (vs the dev branch's cursor over new lines only), which flagged `bridge_just_loaded = true` every 150 ms and emitted `player:position = 0` in a storm. Dev was unaffected because dev had the correct cursor. If the two branches need to *do* the same thing, share one implementation and gate only the logging on `debug_assertions`. The lint at `scripts/check-cfg-debug-assertions.sh` is wired into both release workflows and fails CI on any multi-line `#[cfg(not(debug_assertions))]` block without an explicit `// LINT:cfg-debug-assertions-ok` opt-out comment on the preceding line.
- Other dev/release divergence sources to watch: React PROD vs DEV (StrictMode double-effect only in dev), Vite dev runtime-injected CSS vs release extracted CSS file (insertion-order CSS will diverge), `--release` Cargo optimizations exposing races that dev's slower execution hides.
- **Release diagnostics escape hatch:** `VIBEYTM_LOG_FILE=~/path.log pnpm tauri build` (and the same env var when running the installed `.app` from a terminal via `open -a VibeYTM`) appends tracing output to the file. Use this to diagnose a release-only bug without rebuilding with custom instrumentation. Filter via the standard `RUST_LOG` env var (defaults to `vibeytm=debug,info`). See `src-tauri/src/logging/mod.rs`.

## WKWebView quirks (Tauri visible window)
The visible React UI runs in macOS WKWebView via Tauri. It does not behave identically to Chromium:

- **Click targets MUST be real `<button>` elements.** `<div role="button" tabIndex={0} onClick={...}>` looks a11y-equivalent and passes type-checking, but in this build the synthetic React `onClick` is silently dropped — mouse hover/enter/leave still fires, but the click never reaches React. This regresses every card on Home/Explore/Library/Search at once and is invisible until you actually try clicking. If you must avoid nested `<button>/<button>` HTML, change the INNER element to `<span role="button" onClick={...}>`, never the outer one. Verified 2026-04-24 via diagnostic IPC.
- **`pointer-events: none` on a stale-while-revalidate overlay kills clicks on cached children.** `ReloadOverlay` and similar wrappers must keep children interactive during refresh. Put `pointerEvents: 'none'` on a small corner spinner only, never on the wrapper around the still-visible content. The YTM bridge can stall ~30 s during audio-webview navigation; for that whole window every card becomes click-dead if the overlay blocks events.
- **`ReloadOverlay` MUST visually blur the children during refresh.** A 10 px `filter: blur(10px)` on the children layer is the user-facing "data is reloading" cue. CSS `filter` does NOT affect hit-testing — clicks still pass through to the underlying buttons — so the blur and the no-click-block rule above are independent. An earlier fix removed the blur and the click-block in the same sweep; only the click-block was the bug.
- **NEVER apply `transform` to the children-wrapping layer of `ReloadOverlay`.** `transform: scale(...)` creates a stacking context that this WKWebView build mishandles for hit-testing — clicks on some children stop registering. An attempt to use `scale(0.98)` to hide the blur halo broke clicks across Home / Explore / Library. The blur is enough on its own; transforms on the wrapper are forbidden. Locked in by the contract test in `src/components/LoadingOverlay.test.tsx`.
- **`pointer-events: auto` on a child OVERRIDES the parent's `none`.** If you put `pointer-events: none` on a closed-overlay container (e.g. `NowPlaying` when `isOpen=false`), but ANY descendant has explicit `pointer-events: auto` (e.g. the LRC column when `showLyrics=true`), that descendant remains click-active even though its parent is supposedly inert. Combined with `position: fixed`, this creates an invisible click-stealing region over whatever page is now in front. **Always AND child pointer-events with the parent's open state**: `pointerEvents: isOpen && childIsActive ? 'auto' : 'none'`. Also: when navigating away via the sidebar, reset every overlay flag (`setIsLyricsOpen(false)`, `setIsQueueOpen(false)`, `setIsNowPlayingOpen(false)`) — the sidebar nav handler in `src/App.tsx` does this; new overlays added in the future must too.
- **Background fetches need a settle delay after track change.** YTM's hidden audio webview navigates on every track change, which hangs `fetch()` calls inside it for ~3-15 s. Anything that calls `get_upcoming_tracks` / `get_lyrics` / `next` on track-change should wait ~1.5-2 s before firing so it doesn't pile onto the stuck channel and starve user-driven IPCs (`get_playlist`, `search`) that the user is clicking right then.
- **Seek echoes can arrive 1-4 s late and must not snap state backward.** When the user clicks the progress bar, YTM's bridge sometimes emits stale pre-seek POSITION_UPDATED events while its `<video>.currentTime` is still buffering. The first version of the filter used a fixed 800 ms window — too short — and let echoes through that snapped `useSmoothedPosition` backward, scrolling the lyric panel to a stale line for several seconds. The current rule is: while a seek is pending (set by `markSeek()`), drop every POSITION_UPDATED that's > `SEEK_TOLERANCE_SECS` from the target until either (a) we see a near-target position (clears the pending flag), or (b) `SEEK_RECONCILE_WINDOW_MS` (5 s) elapses (hard cap). The decision logic lives in `src/hooks/seekFilter.ts` as a pure function — keep it pure and add a test in `seekFilter.test.ts` for any new edge case before changing the constants.
- **Bridge `player:track-changed` fires on metadata refinement, not just real track changes.** The poller in `src-tauri/src/webview_bridge/poller.rs` re-emits the track-changed event whenever `duration` grows or `title`/`artist`/`artwork` is refined. The FE's `usePlayerState` filter for stale post-track-change positions must therefore gate `lastTrackChangeAtRef` updates on **`isSameTrack === false`** (videoId actually changed). Treating every emit as a real track change drops every legitimate POSITION_UPDATED for 1.5 s after a metadata refinement — including the post-seek positions, leaving the lyric panel pinned to the pre-seek time. Don't remove the `if (!isSameTrack)` guard around `lastTrackChangeAtRef.current = Date.now()`.
- **User-control state pushed to YTM via IPC must also be persisted to YTM-origin `localStorage` so the bridge can seed it on every page load BEFORE the new `<video>` element exists.** YTM destroys the page context on every track navigation. Globals like `__VIBEYTM_DESIRED_VOLUME_PCT__` are wiped, the prototype-level `volume`/`muted` lock no-ops, and the user hears the new track at default volume for ~one poll cycle until Rust re-pushes the value. The fix is two-sided: (1) in the bridge JS, write the value to `localStorage` on every `set_*` IPC, AND (2) at the very top of `ytm-player-bridge.js` (which Tauri runs as `initialization_script` on every navigation), seed the global from `localStorage` BEFORE the volume lock installs. Any new control surface that has to survive YTM page nav (volume, mute, lyric offset, shuffle, repeat, etc.) needs the same seed-then-persist pair — the IPC alone is too late.
- **`__VIBEYTM_ACCOUNT__` rides the same seed-then-persist pattern (issue #91).** Without it the bridge re-fetched after each navigation with a slightly different avatar URL (DOM-scrape uses `=s96-c`; the API returns the raw signed thumbnails URL), Rust diff fired `player:account-changed`, the React `<CachedImage>` re-decoded, and the user saw a sidebar-avatar flicker on every next/prev. The bridge init also checks the SAPISID cookie — when missing it drops the persisted entry so a sign-back-in doesn't seed the previous user's avatar.
- **First-launch volume seed is `50` (not undefined).** When `localStorage[__VIBEYTM_VOLUME_PCT__]` is absent OR non-finite, seed `__VIBEYTM_DESIRED_VOLUME_PCT__ = 50` so the prototype volume lock clamps from frame zero. Rust's session-restore push one cycle later overrides AND writes a clean localStorage value so subsequent navigations skip the default. Without the seed the user got a brief full-volume burst on the very first track of a fresh install (issue #69).
- **`player:volume` is gated by an emit-on-change guard AND a Rust-side push-settle window — keep both.** The Rust poller previously emitted `player:volume` on every cycle (150 ms). After a YTM page navigation the new `<video>` element transiently reports `volume = 1.0` for 1-2 cycles before the bridge applies our `set_volume` push, and once the frontend's 1200 ms `VOLUME_ECHO_WINDOW_MS` expired that stale 1.0 sailed through and snapped the slider to MAX (issue #76). The fix in `src-tauri/src/webview_bridge/poller.rs` is two halves: (1) only emit when `effective_volume` actually changes (`last_emitted_volume`), and (2) for `VOLUME_PUSH_SETTLE_MS` (2 s) after any `set_volume` push (user IPC OR `bridge_just_loaded` re-seed), trust `stored_volume` over `bs.volume` whenever they disagree by more than 0.01. Removing either half re-opens the bug — track changes at 1200-2000 ms after the last user adjustment hit the gap.
- **Player Next/Previous MUST bypass the planned-queue when shuffle is on.** `PlayerChrome.handleNext`/`handlePrev` consult `getPlannedNext()` / `getPlannedPrevious()` first to give instant optimistic feedback. The planned queue is published by `QueuePanel` from the visible (DOM) queue, which keeps playlist order even with shuffle on — so following the planned-next would defeat shuffle and play the linearly-next track (issue #81). The fix gates the planned-queue lookup behind `!isShuffled`. When shuffle is on, fall through to `playerApi.next()` / `playerApi.previous()` so YTM's internal shuffle cursor picks the random track. Locked in by `src/components/layout/PlayerChrome.test.tsx::"Next bypasses planned queue when shuffle is on"`.

## Lyrics caching + race invariants
- **Cache empty (negative) lyrics results too — don't only persist hits.** `src-tauri/src/commands/browse.rs::get_lyrics` writes the result to disk even when `text` is empty AND `lines` is empty (issue #74). The lyrics cache has no time-based TTL — the negative entry sticks around until LRU eviction or explicit refresh. The Refresh button in Now Playing bypasses the cache via `force_external`, so a re-indexed LRCLIB hit can still surface on demand. Without this, every replay of a lyric-less track re-ran the YTM → LRCLIB → NetEase pipeline. Locked in by `src-tauri/src/cache/mod.rs::tests::lyrics_cache_persists_negative_results`.
- **External lyric race runs in PARALLEL with YTM's `browse?browseId=<lyricsId>` (issue #75).** `spawn_external_lyrics_race` spawns LRCLIB+NetEase as a `tokio::task::JoinHandle` BEFORE the YTM browse fetch. When YTM returns synced lyrics, the spawned task is `abort()`ed (JoinHandle does NOT auto-cancel on drop). The instrumental short-circuit must also abort. `?` exits on browse failure intentionally don't abort — the spawned task is bounded by reqwest timeouts.
- **UGC titles need the split-and-fanout candidate set (issue #64).** `clean_query_field` cuts at `｜` / ` | ` / ` - ` (destructive); `clean_query_preserve_separators` keeps them so `ugc_split_title_artist` can recover both halves of an `Artist - Title` packed string. The race fans out across `[(clean_artist, clean_title), (split_left, split_right), (split_right, split_left)]` so LRCLIB/NetEase can match either orientation. Bare ASCII `-` only counts as a delimiter when both halves are whitespace-free AND at least one side has non-ASCII characters (CJK heuristic) AND there's exactly one dash in the input.
- **Short-title matches require exact normalized equality (issue #67).** `title_matches` falls into an exact-equal branch when the normalized request is ≤4 chars (e.g. "APT." → "apt"). Substring matching alone let "Adapt"/"Aptitude" win against duration tolerance. `lyrics_normalize` strips ASCII punctuation + collapses whitespace; locked in by 5 unit tests in `ytm_api::tests`.
- **UGC cover art falls back to iTunes Search (issue #65).** `useExternalCoverFallback` fires only when the bridge artwork is non-album-art AND has artist/title; gated behind `BRIDGE_SETTLE_MS` to respect the bridge-channel rule, and a per-videoId failure cache prevents dep-array re-fires from replaying a failed lookup. Apple Music's `is*-ssl.mzstatic.com` is whitelisted in `isAlbumArtUrl` so cached results don't re-trigger the fallback.

## Bridge same-track snapshot consistency (issue #57)
- **The same-videoId merge branch in `webview_bridge/poller.rs` MUST gate on a self-consistent snapshot.** When the user seeks to the very end of a short track YTM auto-advances mid-poll. The bridge briefly reports the OLD videoId alongside the NEW track's duration (or vice versa). The previous merge wrote the new duration onto the old track → FE rendered new-duration + old-cover with stalled playback. Fix: require `bs.video_id == t.video_id` AND `!bs.video_id.is_empty()` before merging any field deltas. The single read-lock pattern (one `player_state.read().await` for both the consistency check and the delta evaluation) prevents TOCTOU.

## QueuePanel covers (issue #68 + #96)
- **Always fetch /next for upcoming-row artwork overrides — don't skip when liveQueue contains the current track.** The DOM-scraped queue typically carries `i.ytimg.com` video thumbnails (filtered by `isAlbumArtUrl`); without the `/next` enrichment, song-radio queues render placeholders for every upcoming row. The per-(videoId, activePlaylist) `queueCache` short-circuits the network call after the first play.
- **`setFetchedUpcoming([])` flush ONLY on activePlaylist change.** A track-change inside the same playlist must NOT flush — that briefly empties artwork overrides for the upcoming rows during the BRIDGE_SETTLE_MS+network window. Genuine playlist switches DO flush so wrong-track artwork doesn't pin to the new playlist's rows.
- **`QueueArtwork`'s per-row `useAudioCounterpartArtwork` lookup is gated behind `BRIDGE_SETTLE_MS`.** Without the gate a 10-row queue panel opened during a track change fans out 10 simultaneous IPCs and starves user-driven calls. The hook itself is module-cached and de-duped across rows so steady-state cost is one IPC per uncached videoId.

## Frontend logging
- **Use `debug.error(tag, ...)` from `src/lib/debug.ts` for catch-handler logging in components/pages, NOT raw `console.error`.** `debug.error` is gated behind the `vibeytm:debug` localStorage flag and pipes to the Rust dev-server terminal via the `debug_log` IPC, so users can flip it on (`vibeytmDebugOn()` in WebView devtools) and see frontend errors in the same place as Rust logs. The three top-level handlers in `src/main.tsx` (`window.error`, `window.unhandledrejection`, React error boundary) are the documented exception — they MUST stay on raw `console.error` so crash-recovery paths surface even when debug is off.

## Playlist mutations + library auto-refresh
- **`TrackInfo.setVideoId` is REQUIRED for `remove_track_from_playlist`.** YTM's `ACTION_REMOVE_VIDEO` rejects calls that pass only `removedVideoId` — it needs `setVideoId` too because a video can appear multiple times in the same playlist. The field is only populated by `parse_track_from_list_item` in `src-tauri/src/ytm_api/mod.rs` (playlist + album detail rows carry `playlistItemData.playlistSetVideoId`). Search results, bridge-sourced tracks, and queue rows DO NOT carry it. `PlaylistDetailPage` gates the Trash2 hover affordance behind `track.setVideoId && track.videoId` so non-removable rows quietly omit the button instead of producing a confusing "couldn't remove" error.
- **`add_track_to_playlist` returns `bool` — `true` = added, `false` = YTM deduped.** The Rust helper `parse_add_track_response` inspects `actions[*].addedVideoIds` for the requested videoId. Toasts MUST branch on this signal: `Added to <name>` vs `Already in <name>`. Skipping the branch makes the UX lie when the user re-adds a track that's already in the playlist (YTM's `DEDUPE_OPTION_SKIP` makes that a no-op server-side, but the user still sees `Added to …` and assumes a duplicate was inserted). Locked in by 5 unit tests on `parse_add_track_response` and one picker integration test.
- **`notifyLibraryMutated()` (`src/lib/libraryMutations.ts`) is the ONE signal that auto-refreshes `LibraryPage`.** Any new mutation surface (rename, reorder, move, duplicate, …) MUST call it after success. App.tsx subscribes via `subscribeToLibraryMutations` and bumps `libraryVersion`, which is the prop `LibraryPage` keys its refetch on. `LibraryPage` stays mounted across detail-page nav, so without the bump a deleted playlist still appears in the grid on the way back. Don't try to refetch directly from the mutator — the registry is the contract.
- **`AddToPlaylistPicker` subscribes to `subscribeToLibraryMutations` at module load so its 60s playlist cache is invalidated by EVERY mutation surface, not just its own add/create/delete.** Otherwise removing a track from `PlaylistDetailPage` (which only calls `notifyLibraryMutated`) would leave the next picker open showing the pre-remove `trackCount` for up to a minute. The pattern: track-count-bearing caches that aren't owned by `LibraryPage` itself must hang off `subscribeToLibraryMutations` for the same reason.
- **The picker and toast are root-mounted singletons.** `<AddToPlaylistPicker />` and `<Toast />` are mounted exactly once in `App.tsx`. The picker is triggered via `openAddToPlaylistPicker({ videoId, trackTitle, position })`; subsequent calls re-anchor the same dialog via `key={request.videoId}`. The toast is triggered via `toast.show({ message })`; subsequent calls replace the current toast (no stacking). Rendering either component anywhere else creates duplicate state machines that fight over the active request.
- **`SongRow`'s hover quick-actions are `<span role="button">` not `<button>`.** The row itself is a `<button>`, and WKWebView silently drops synthetic `onClick` for nested-button children (same rule as the WKWebView quirks section). When adding new inline row actions (the destructive `danger` variant for `Remove from playlist` is the current pattern), reuse `InlineAction` — don't reach for a real `<button>`.

## Bridge IPC dedup — idempotence for state-push handlers (issue #106)
- **`on_track_changed` MUST dedup against the stored track before emitting.** The bridge JS (`scripts/inject/ytm-player-bridge.js::update`) is called from `setInterval(update, 150)` and invokes `on_track_changed` on every tick — NOT only on real change. Without the dedup, the FE receives ~6.7 `player:track-changed` events per second; six `usePlayerState` instances each `setState` → six render cascades → `useSmoothedPosition`'s rAF baseline gets perturbed on every tick (progress-bar flicker) and `LyricsPanel`'s active-line scroll effect re-fires (lyric jitter). The Rust handler at `src-tauri/src/commands/player.rs::on_track_changed` short-circuits when `track_bridge_fields_equal(stored, incoming)` returns true. Compare ONLY the fields the bridge actually populates (`video_id`, `title`, `artist`, `album`, `artwork_url`, `duration_secs` ±0.5 s tolerance) — the stored track may carry richer metadata from `play_track` (`artist_id`, `set_video_id`, …) that the bridge can't replicate, so a full struct-equality check would falsely report "changed" every tick.
- **The bridge JS ALSO dedups via `window.__VIBEYTM_LAST_IPC_TRACK__`.** Defense in depth — saves the IPC marshaling cost on the hot path so the Rust handler isn't invoked 6.7×/sec just to be dropped.
- **General rule for `on_*` IPCs that the bridge can call on an interval**: treat them as state-push (idempotent), not as one-shot change events. Compare against stored state and skip emit when identical. The Rust poller already follows this pattern (`last_emitted_position`, `last_emitted_volume`, `last_video_id`, `last_status`); IPC handlers MUST mirror it. Locked in by `track_bridge_fields_equal_*` unit tests in `commands::player::tests`.

## Liquid Glass token system (`src/styles/tokens.css`)
- **`--glass-recipe` is the canonical `backdrop-filter` value.** All glass surfaces (popovers, dialogs, chrome plates, settings cards) MUST set `backdrop-filter: var(--glass-recipe); -webkit-backdrop-filter: var(--glass-recipe);` and nothing else. The recipe bundles `blur + saturate + brightness`; using `blur(...)` alone (the previous failure mode in `UpdateBanner`, `ShortcutCheatsheet`, `ContextMenu`, `Sidebar`) desaturates the glass to gray on dark UIs — what the tokens.css comment block calls "the ambient page colour passes through, matching the uniform liquidGL look."
- **`--glass-recipe-strong` is for full-screen scrims** (`NowPlaying`, `FocusTimer`, focus-confirm modal, `Sidebar` while NowPlaying is open). Tighter blur (40 px) tuned so the underlying art isn't over-diluted.
- **`--glass-plate-shadow` and `--glass-card-shadow` are the only sanctioned `boxShadow` values for popovers and cards.** Plate (rim + heavy 24 px / 60 px drop) goes on `AddToPlaylistPicker`, `LyricsOverlay`, `QueuePanel`. Card (rim + soft 2 px / 12 px drop) goes on `SettingsCard`. Hand-rolling `'inset 0 1px 0 var(--glass-rim-bright), 0 24px 60px oklch(0% 0 0 / 0.5)'` inline duplicates a token that already exists — when a new popover ships, reach for the token first.
- **Tile recipe untouched.** Active-state tiles (HomePage greeting card, ExplorePage mood pill, SearchPage tab, SongRow/EpisodeRow hover, picker NewPlaylist row) continue to use `var(--glass-tile-bg[-active])` + `var(--glass-tile-shadow[-rest])`. No `backdrop-filter` on these — they sit inside an already-blurred parent and stacking filters triggers issue #99's WKWebView paint feedback loop.
