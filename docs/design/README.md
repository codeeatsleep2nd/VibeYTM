# VibeYTM 2.0 — design + planning docs

The complete planning trail that produced the SwiftUI v2.0 rewrite.
Generated by the [gstack](https://garryslist.org/) skill workflow:

```
/office-hours
    ↓
/plan-eng-review  (11 decisions, including outside-voice review by Codex)
    ↓
/plan-design-review  (10 decisions, all 7 review passes)
    ↓
/prp-plan + /prp-implement  (Sprint 0 build)
```

These are kept in the repo (not just in `~/.gstack/projects/`) so the
decision trail stays grep-able for future contributors. If you're picking
up Sprint 1+ work, read these first.

## Files

| File | What it is | Source skill |
|---|---|---|
| [`dongli-SwiftUI-design.md`](dongli-SwiftUI-design.md) | The design doc — problem statement, 4 premises, 3 alternatives considered, the 6-sprint roadmap, every architectural decision (D1-D15 across all review passes), failure modes, "NOT in scope" list, design decisions added by `/plan-design-review`, reviewer concerns | `/office-hours` + `/plan-eng-review` + `/plan-design-review` |
| [`dongli-SwiftUI-eng-review-test-plan.md`](dongli-SwiftUI-eng-review-test-plan.md) | Test coverage plan — affected surfaces, key interactions to verify, edge cases, critical paths, test files to create per sprint | `/plan-eng-review` |

| [`sprint-0-spine-report.md`](sprint-0-spine-report.md) | Sprint 0 implementation report — tasks completed, deviations from plan, validation results, files changed | `/prp-implement` |

## How to read these in order

If you're new to the v2.0 plan, read them roughly in this order:

1. **Design doc** — start at the "Problem Statement" and skim down to "Approaches
   Considered." This gives you the "why are we doing this" framing in ~5 minutes.
2. **Design doc → "Recommended Approach"** — has the 6-sprint roadmap. Skim
   Sprint 0 in detail; the other sprints are summary-level.
3. **Design doc → "Open Questions" + "RESOLVED" lines** — the decisions made
   and the ones still open.
4. **Sprint 0 PRP** — the actual file-by-file implementation contract.
5. **Sprint 0 report** — what actually happened vs. what was planned.
6. **Test plan** — when you're about to implement, this tells you what tests
   to write per surface.

## What's NOT here (yet)

PRP plans for Sprints 1-5 weren't generated separately — the implementation
worked directly off the design doc's per-sprint checklists. If you want a
proper PRP for Sprint 1+ before continuing, run:

```
/prp-plan "Sprint N from docs/design/dongli-SwiftUI-design.md"
```

This will produce `sprint-N-...plan.md` in the same format as the Sprint 0
PRP.

## When this folder can be cleaned up

These docs are archival reference. They never become stale per se — they
document decisions made on a specific date. When v3.x lands, write a new
design doc; don't edit these. They're a record of how the v2.0 thinking
unfolded.
