## Problem Statement

VibeYTM ships today on `main` as Tauri 2.x + React 19 + Rust (v1.5.0). The
visible UI runs in WKWebView, which has produced ~30 documented workaround
patterns in CLAUDE.md (click-target swallowing on `<div role="button">`,
`pointer-events: none` overlay traps, `transform` breaking hit-testing,
stale-seek echoes, bridge saturation during track-change navigations, volume
push-settle windows, etc.). Each individual fix is a one-off — the underlying
WebView platform keeps generating new variants of the same class of bug. The
chrome also doesn't feel native: every Liquid Glass surface is a CSS shim
faking what macOS 26's `NavigationSplitView` / `.glassEffect()` give you for
free, and there's no path to true macOS-native features (AppIntents, Spotlight
entities, WidgetKit, Foundation Models) from inside Tauri.

A SwiftUI rewrite has been in progress on the `SwiftUI` branch for several
weeks. The hard architectural ports are done: a hidden-WKWebView audio engine
(YTMBridge), the three pure-logic quirk modules (`SeekFilter`, `VolumeSettle`,
`TrackChangeGuard`) with Swift Testing suites, `PlayerCore` as a typed value
model, `BridgeReducer` composing the pipeline, `MPNowPlayingInfoCenter` /
`MPRemoteCommandCenter` wired for media keys + Now Playing widget. Most views
exist (Home, Search, Library, BrowseDetail, NowPlayingExpanded, LyricsPanel,
QueuePanel, PlayerChrome, SidebarView, AuthWebView, SettingsView). About 60-70%
of `main` is ported.

What "rewrite everything" actually means: **finish the existing SwiftUI port to
core-loop parity, add the four macOS-26-native features that justify going
native in the first place, then delete the Tauri tree.** Not throw away the
50 Swift files already on disk.

## What Makes This Cool

A pretty SwiftUI port of an existing app is forgettable. The version worth
shipping is the one Codex named in its second-opinion read:

> A true on-device "DJ copilot," not just a prettier player.

That means a `QueuePlan` / `SessionPlan` modeled with `@Generable`, run through
`SystemLanguageModel.default` + `LanguageModelSession.prewarm()`, given `Tool`s
that wrap the existing `Innertube` search + the bridge's `play(item:)` /
`like` / `queue` actions. Exposed as an `AppShortcut`, openable from
Spotlight, surfaceable in a Mac desktop widget and a Control Center music
tile. Killer prompt: *"Keep this vibe, but darker, no live versions, and land
on something huge in 25 minutes."*

That's the version where someone goes "wait, you can do that?" — not "oh, a
SwiftUI port." Everything else in this doc exists to make that one feature
shippable.

## Constraints

- **Single developer**, weekend cadence. ~6-10 weeks of intermittent effort.
- **macOS 26 (Tahoe) is the floor.** Loses every Mac user not on Tahoe.
  Accepted because Liquid Glass + Foundation Models + the new AppIntents
  surface require it.
- **Hidden WKWebView audio engine is permanent.** YTMBridge.swift continues
  to load music.youtube.com and poll `window.__VIBEYTM_STATE__` every 150 ms.
  AVPlayer + extracted itag URLs explicitly rejected (loses Premium / DRM /
  catalog, ongoing arms race with YouTube).
- **Swift 6.2 strict concurrency.** No `@unchecked Sendable` shortcuts; the
  existing packages already build clean under strict concurrency.
- **Xcode 26 required** for build (Liquid Glass APIs, the new AppIntents,
  Foundation Models, ControlWidget).
- **No iPhone companion app.** Therefore no `ActivityKit` Live Activities —
  Codex's correction. Mac surfaces only: desktop widgets, Control Center,
  AppShortcuts.

## Approaches Considered

### Approach A: Spine-First Cutover (chosen)

Sprint 0 lifts architectural foundation. Sprints 1-2 finish core-loop parity.
Sprints 3-4 add macOS-26-native features in dependency order (chrome first,
then AppIntents/Spotlight, then Foundation Models + widgets). Sprint 5
mops up post-beta features (FocusTimer, History, tray, updater). Sprint 6
deletes the Tauri tree and ships v2.0 stable.

Effort: XL — realistic floor is **12-16 weekend-effort weeks** for a
single developer (the initial 6-10 estimate underweights first-time
Foundation Models tool-calling, two widget targets, signing/notarization
plumbing, and the README/CLAUDE.md rewrites).
Risk: Low. Every feature is built once on a stable spine.

### Approach B: Feature-First Cutover

Finish missing views first, ship 2.0-beta on the current SPM shell, retrofit
Xcode project + App Group + AppRouter after. Faster to beta (~2-3 weekends),
but every macOS-26 feature requires re-architecting state ownership AFTER
beta users already exist.

Effort: L. Risk: Medium (you migrate the spine mid-life of v2.x).

### Approach C: Two-Track Parallel

Spine and parity work alternate weekends. No clear "done" signal.
Realistic risk: parity track ships, spine track never does, beta lands on
the dev-launch hack anyway.

Effort: L-XL. Risk: Medium.

      let description = "Play a track by videoId in the YTM bridge."
      @Generable
      struct Arguments {
          @Guide("YouTube videoId, 11 characters")
          var videoId: String

      let bridgeHost: BridgeHost

      func call(arguments: Arguments) async throws -> ToolOutput {
          // Direct await — Swift hops to MainActor automatically because
          // BridgeHost.play(videoId:) is @MainActor async. No explicit
          // MainActor.run wrapper needed (and using one with an async
          // body is the wrong shape — MainActor.run is for hopping a
          // synchronous block to the main actor).
          try await bridgeHost.play(videoId: arguments.videoId)
          return ToolOutput("Started playing \(arguments.videoId)")

  }

#### 4b — Host ↔ extension IPC (architectural — must precede widgets)

Widgets and Control Center extensions are **separate processes** and cannot
touch the host's `WKWebView`. The AppIntents they fire need a path back to
`BridgeHost` running in the host process. Pattern:

1. The extension's intent (`PlayPauseIntent`, `NextTrackIntent`, etc.)
   appends a "command record" (one JSON object per line — NDJSON:
   `{action, args, requestedAt}`) to a known file in the App Group
   container under an `NSFileCoordinator` write coordination block, then
   posts a Darwin notification `com.vibeytm.app.command-pending`.
   NDJSON + append-only handles two extensions firing simultaneously
   without losing either command.
2. The host app subscribes to the Darwin notification at launch via
   `CFNotificationCenterAddObserver`. On fire, it drains the command
   file under an `NSFileCoordinator` write block: read all records,
   truncate the file to zero, then dispatch records in `requestedAt`
   order to `BridgeHost`. Drops any record older than 10 s (stale GC —
   prevents replay if the host was force-quit mid-drain). The intent's
   `perform()` resolves once it sees the snapshot update reflecting the
   command (or after a 2 s timeout).
3. For "open the app at X" intents (`OpenAlbumIntent`), use the URL scheme
   from Sprint 3 — `vibeytm://album/{browseId}` — which launches the host
   if not running and routes through `AppRouter` either way.

**Drainer spec (full).** Three implementation details the high-level
flow above leaves out:

- **Isolation hop.** `CFNotificationCenterAddObserver`'s callback fires on
  a CF run loop, not `@MainActor`. The observer callback wraps the drain
  work in `Task { @MainActor in ... }` so `BridgeHost` (MainActor-bound)
  can be called safely. Latency cost: one MainActor hop (~1 ms).
- **Command file lifecycle.** The file at `<AppGroup>/commands.ndjson` is
  created lazily by the first extension write. Atomic-append between
  cooperating processes is enforced by `NSFileCoordinator` (the
  cross-process file-locking primitive macOS provides — PIPE_BUF only
  governs pipes/FIFOs, not regular files, so don't rely on it here).
  Every extension write wraps the append in
  `NSFileCoordinator(filePresenter: self).coordinate(writingItemAt: url, options: [], error: &err) { coordinatedURL in ... }`;
  the host's drain wraps the read-and-truncate in
  `coordinate(writingItemAt:options: [.forDeleting], ...)`. Cooperative
  locking means simultaneous writers block, not corrupt. Host on launch
  ensures the App Group directory exists and has the right permissions;
  the file itself is created on demand.
- **Host-running detection via heartbeat.** Host writes the current Unix
  timestamp to `<AppGroup>/heartbeat` every 5 s (`Timer.scheduledTimer`
  on MainActor). AppIntent `perform()` reads the heartbeat; if the value
  is older than 10 s, it returns a user-facing error via the intent's
  thrown `IntentError` with localized message "VibeYTM isn't running —
  open the app and try again." Widgets show the snapshot regardless
  (it's just stale data from disk), but the transport buttons surface
  the error in the widget's interactive feedback.

### DJ Copilot "Vibe" sheet

- **UX shape (D2)**: SINGLE TEXT FIELD, ChatGPT-style. Placeholder
  "Describe the vibe…". Three example prompts shown below in ghost text:
  "darker, no live versions, 25 minutes" / "90s alt-rock workout" / "keep
  this going, mellower". Recent-prompts strip below the examples (last 3
  prompts as glass chips, tap-to-refill).
- **Entry point (D2)**: Pill button in PlayerChrome next to Like / Shuffle
  / Repeat. Always visible while a track plays. The plan's `AppShortcut`
  + `vibeytm://vibe` URL are secondary entry points.
- **Loading state (D3)**: STREAMING. Sheet header changes to "Building
  your vibe…". Queue panel auto-opens in the background. As each
  `TrackSuggestion` streams from the model, it animates into the queue
  with a soft fade-in (Reduce-Motion-respecting: skips animation, instant
  insert). Sheet auto-dismisses when stream completes. Implementation:
  use Foundation Models' `@Generable` streaming-decode (the framework
  emits partial QueuePlans as tokens arrive).
- **Empty / error state (D4)**: Inline fallback. Empty model result → sheet
  shows "Couldn't build that vibe. Search YTM for it instead?" with a
  Search button that runs the prompt through `Innertube.search()` and
  navigates `AppRouter` to `.search(query: prompt)`. Foundation Models
  unavailable (entitlement denied or model load failure) → same recovery
  path with copy "AI unavailable. Search YTM for your prompt instead?"
- **Post-Vibe playback (D6)**: Current track CONTINUES; only the upcoming
  queue is replaced. When the current track finishes, `auto-advance` plays
  `vibePlan.tracks[0]`. Never interrupt mid-song.
- **Icon (D7)**: Single YouTube-red `sparkles` SF Symbol tinted with
  `--color-accent` (OKLCH 65% 0.20 25). NOT the Apple Intelligence rainbow
  gradient. On-brand, distinct from every other AI-bolted-on app.

## What I noticed about how you think

- You picked all four macOS-26 native features in D1 without negotiating
  scope. Most people would've cut Foundation Models or Widgets to look
  "realistic." You didn't — and Codex caught the same thing: *"That says
  the excitement is not 'replace Tauri.' It's 'build the absurdly native
  Tahoe music app.'"* Hold on to that. The kitchen-sink version is the
  version worth shipping; aim for the kitchen-sink version even if you
  end up shipping 80% of it.
- You named the audio engine as "permanent" without flinching. That's a
  sign you understand which battles to fight: the WKWebView quirks are
  isolatable (SeekFilter, VolumeSettle, TrackChangeGuard already prove
  that), but the YTM auth/DRM/catalog dependency is not — and you're not
  pretending otherwise.
- You said yes to deleting the Tauri tree at v2.0 instead of keeping it
  as a fallback. That's the right call but also unusual — most people
  hedge with "let's keep both for safety." Conviction beats hedge.
- You agreed to all three premise challenges in D5 without revising any
  of them — and then in D7 you accepted Codex's pushback on Live
  Activities immediately. That mix (firm on what you've thought through,
  flexible when shown new information) is the operating mode that
  actually finishes rewrites instead of abandoning them.
